## 원문 요약

이 글은 기존 웹 분석 도구가 사용자가 무엇을 했는지는 보여주지만 왜 그렇게 행동했는지 설명하는 데 한계가 있다고 전제한다. Google Analytics 같은 정량 분석 도구만으로는 이탈, 무시된 버튼, 좌절, 혼란, 동기 같은 맥락을 충분히 이해하기 어렵고, 전통적인 UX 리서치는 대량의 실시간 상호작용 데이터를 처리하는 데 제약이 있다고 설명한다.

Microsoft Clarity는 히트맵, 세션 레코딩, 자동화된 인사이트를 무료로 제공해 정성적 행동 분석의 비용과 복잡성 장벽을 낮추는 도구로 소개된다. 글은 클릭/스크롤/영역 히트맵, 세션 레코딩, 인사이트 대시보드를 통해 사용자 여정의 병목을 찾고, 레이지 클릭, 데드 클릭, 과도한 스크롤, 퀵 백 같은 semantic metrics를 전환 실패의 선행 지표로 활용할 수 있다고 정리한다.

CRO 전략 측면에서는 GA4 같은 정량 분석으로 문제가 큰 페이지나 퍼널 단계를 찾고, Clarity의 히트맵과 레코딩으로 왜 문제가 생기는지 확인한 뒤, 개선 가설과 A/B 테스트로 이어지는 반복 workflow를 제안한다. 또한 Above the Fold 최적화, 폼 제출 최적화, 카트 이탈 감소, 모바일 경험 개선 사례를 들어 Clarity가 UX 개선뿐 아니라 CPA, 전환율, 수익성 개선과 연결될 수 있다고 주장한다.

## 핵심 주장

- 정량 분석은 행동의 규모와 위치를 보여주지만, 사용자 좌절과 혼란의 맥락을 직접 설명하는 데 한계가 있다.
- Clarity의 히트맵과 세션 레코딩은 사용자의 실제 행동을 시각적으로 관찰하게 해 UX 병목을 찾는 데 유용하다.
- 레이지 클릭, 데드 클릭, 과도한 스크롤, 퀵 백 같은 semantic metrics는 전환 실패 이전에 나타나는 선행 지표로 볼 수 있다.
- GA4와 Clarity는 대체 관계보다 보완 관계로 쓰는 편이 적합하다. 정량 데이터로 문제 구간을 좁히고, 정성 데이터로 원인을 관찰하는 흐름이 좋다.
- Clarity는 privacy masking과 낮은 성능 오버헤드를 통해 상시적 UX 최적화 인프라로 활용될 수 있다고 글은 주장한다.

## 내 프로젝트에 주는 시사점

- LLM/agent 기반 제품이나 wiki UI를 만들 때, 단순 page view나 클릭 수보다 사용자가 어디서 막히는지 관찰하는 정성적 행동 데이터가 중요하다.
- 제품 PoC의 초기 단계에서는 GA4류 정량 지표와 Clarity류 세션 관찰을 함께 두면 UX 개선 우선순위를 더 빨리 잡을 수 있다.
- AI-native workflow에서도 "자동화가 산출물을 만들었다"는 사실만으로 충분하지 않다. 사용자가 실제로 어디서 혼란을 겪는지 관찰하고, reviewer나 human-in-the-loop 지점을 조정해야 한다.
- 세션 레코딩 도구는 개인정보와 민감 정보 노출 위험이 있으므로 masking, consent, data retention 정책을 먼저 확인해야 한다.

## 검증 필요 주장

- Clarity가 무료로 제공하는 기능 범위, retention, sampling, export, pricing 조건은 Microsoft 공식 문서에서 재확인해야 한다.
- 글에 제시된 전환율 증가, CPA 감소, 카트 포기율 감소, 모바일 세션 포기율 감소 수치는 사례별 맥락과 원 출처 확인이 필요하다.
- Clarity의 성능 오버헤드가 낮다는 주장은 실제 사이트, script 설정, 네트워크 조건에 따라 다를 수 있다.
- PII masking이 모든 민감 정보 노출을 자동으로 막는 것은 아니므로 프로젝트별 설정 검증이 필요하다.
