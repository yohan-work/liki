## 원문 요약

이 PDF는 AI-native startup의 lifecycle을 Idea, MVP, Launch, Scale 네 단계로 재구성한다. 공통 메시지는 AI가 build, research, documentation, automation을 압축하지만 founder의 판단을 대체하지 않는다는 것이다. AI가 기술 병목을 줄인 만큼, 검증 없는 구축, premature scaling, confirmation bias, agentic technical debt, insecure MVP, founder bottleneck 같은 실패 패턴이 더 빠르게 발생할 수 있다고 경고한다.

## 핵심 주장

- founder의 역할은 직접 실행자에서 agent, tool, workflow를 조율하는 orchestrator로 이동한다.
- Idea 단계의 exit criteria는 real human conversation 기반의 problem-solution fit이다.
- MVP 단계의 exit criteria는 retention, revenue, referral 같은 product-market fit 신호다.
- AI-generated MVP는 scope document, architecture context, session log, security review, measurement framework가 없으면 기술 부채가 빠르게 누적된다.
- Launch 단계에서는 founder attention을 operational system으로 대체하고, technical debt, security, compliance, product management process를 workstream으로 관리해야 한다.
- Scale 단계에서는 domain expertise, institutional knowledge, user behavior data, integration depth가 moat로 전환될 수 있다고 본다.

## 내 프로젝트에 주는 시사점

LLM Wiki의 idea와 MVP page에도 "빌드 가능한가"보다 "검증했는가"와 "다음 stage로 넘어갈 근거가 있는가"를 먼저 써야 한다. 특히 개인 PoC는 AI coding으로 빠르게 만들 수 있으므로 scope creep과 false positive를 더 엄격하게 관리해야 한다. `AI 트렌드 레이더 7일 PoC` 같은 항목은 MVP 기능 목록보다 사용자 반복 사용, 추천, 유지율, 수동 follow-up 필요도 같은 evidence metric을 먼저 정해야 한다.

## 검증 필요 주장

- "42% of startups failed because they built something nobody wanted" 수치는 자주 인용되는 통계지만 원 출처와 현재성 검증이 필요하다.

- Claude Cowork, Claude Code Security, MCP connector, Agent SDK 관련 기능은 2026-06-18 현재 실제 availability와 정책 확인이 필요하다.
