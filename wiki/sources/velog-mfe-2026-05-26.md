## 원문 요약

이 글은 마이크로프론트엔드(MFE)를 큰 프론트엔드 애플리케이션을 독립적으로 개발, 배포, 운영 가능한 작은 단위로 나누는 아키텍처로 설명한다. 문제 배경으로는 모놀리식 프론트엔드에서 발생하는 전체 재빌드/재배포 의존성, 팀 간 머지 충돌, 기술 스택 통일 제약, 빌드/테스트 시간 증가를 든다.

저자는 단순 폴더 구조 분리나 모듈 분리만으로는 MFE가 해결하려는 독립 생명주기 문제가 충분히 풀리지 않는다고 설명한다. 핵심 판단 기준은 "언제 통합할 것인가"와 "어디서 통합할 것인가"다.

통합 시점으로는 빌드 타임 통합과 런타임 통합을 비교한다. 빌드 타임 통합은 각 앱을 npm package나 Nx/Turborepo 같은 monorepo package로 관리한 뒤 최종 bundle에서 합치는 방식이며, 코드 일관성, 초기 로딩, 타입 안정성의 장점이 있지만 빌드 시간과 library version 통일 문제가 남는다. 런타임 통합은 독립 배포된 앱을 사용자가 접속할 때 조립하는 방식이며, Webpack 5 Module Federation의 host/remote 구조, singleton 설정, Next.js Multi-zone rewrites를 예로 든다.

통합 위치로는 SSR, CSR, edge computing을 다룬다. SSR은 초기 로딩과 SEO에 유리하지만 서버 부하가 생길 수 있고, CSR은 동적인 UX에 유리하지만 초기 로딩과 SEO가 불리할 수 있으며, edge computing은 사용자 근처에서 앱을 조합해 응답 시간을 줄일 수 있지만 인프라와 비용 부담이 있다고 설명한다.

마무리에서는 MFE가 큰 조직과 큰 애플리케이션에서 팀 분리, 독립 배포, 기술 스택 분리의 장점이 있지만 코드 중복, UX 일관성, 리소스 증가, 조직의 기술 이해도 같은 새로운 복잡성을 만든다고 정리한다.

## 핵심 주장

- MFE는 프론트엔드 애플리케이션을 독립 개발/배포/운영 가능한 단위로 나누는 아키텍처다.
- 단순 코드 분리는 MFE와 다르며, 각 단위의 독립 생명주기와 통합 방식이 중요하다.
- 빌드 타임 통합은 일관성과 초기 성능에 유리하지만 독립 배포와 버전 분리 이점은 제한적이다.
- 런타임 통합은 독립 배포와 장애 격리에 유리하지만 library version, singleton, 상태 공유, 초기 로딩 문제가 생길 수 있다.
- MFE 도입은 유행이 아니라 조직 규모, 팀 역량, 배포 병목, 기술 스택 분리 필요성을 기준으로 판단해야 한다.

## 내 프로젝트에 주는 시사점

개인 PoC나 작은 프로젝트에서는 MFE가 대체로 과한 선택일 수 있다. 그러나 여러 frontend surface를 독립 배포하거나, Codex/agent가 서로 다른 앱 단위를 병렬로 수정하는 실험을 한다면 MFE의 장점과 복잡성을 판단 기준으로 삼을 수 있다. 특히 "코드 분리"와 "독립 생명주기"를 혼동하지 않는 것이 중요하다.
