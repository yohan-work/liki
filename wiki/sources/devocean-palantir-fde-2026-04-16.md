# DEVOCEAN Palantir FDE Article

## 요약

DEVOCEAN 글 "Palantir FDE: 소프트웨어를 '만드는 것'에서 '작동시키는 것'으로의 전환"이다. 글은 AI와 데이터 플랫폼 PoC가 성공해도 실제 조직 운영에는 변화가 생기지 않는 문제를 출발점으로 삼고, Palantir의 Forward Deployed Engineer(FDE)를 복잡한 시스템을 현장 운영에 연결하는 역할이자 운영 패턴으로 설명한다.

글의 핵심은 FDE를 단순한 개발자나 컨설턴트가 아니라 고객 조직에 임베드되어 problem framing, data integration, rapid prototyping, productionization, adoption and scaling까지 end-to-end로 책임지는 execution layer로 보는 것이다.

## 원문이 말한 핵심 주장

- AI와 데이터 플랫폼 프로젝트는 모델 정확도, 데이터 충분성, PoC 성공에도 불구하고 현실의 운영 변화로 이어지지 않을 수 있다.
- 실패의 핵심 원인은 기술 한계보다 개발, 데이터, 비즈니스 의사결정을 실제 운영 시스템으로 연결하는 역할의 부재다.
- FDE는 고객 조직 내부에 임베드되어 데이터 통합부터 애플리케이션 개발, 운영 적용까지 수행하며 비즈니스 임팩트를 만드는 역할이다.
- Dev는 "one capability, many customers"에 가깝고, FDE는 "one customer, many capabilities"에 가깝다.
- FDE의 execution pipeline은 problem framing, data integration, rapid prototyping, productionization, adoption and scaling으로 정리된다.
- FDE는 데이터, 플랫폼, 애플리케이션, 비즈니스 프로세스 사이의 glue layer 역할을 한다.
- Palantir 모델은 제품만 제공하는 SaaS나 인력만 제공하는 컨설팅과 달리 제품과 FDE를 결합한 go-to-market 전략으로 설명된다.
- 성공은 기술 자체보다 adoption에서 결정되며, 데이터 거버넌스, 조직 반발, vendor dependency는 실패 요인이 될 수 있다.

## 위키에 주는 시사점

- [[Forward Deployed Engineer]]는 [[Agentic Workflow]]와 [[ML Model Serving Pipeline]]을 실제 조직 운영에 붙이는 역할 모델로 볼 수 있다.
- LLM Wiki의 PoC도 "잘 만든 자동화"보다 "반복 루틴에 실제로 들어가 쓰이는 자동화"를 성공 기준으로 삼아야 한다.
- [[Personal Operating System]]에서는 source capture, triage, execution, adoption, review가 분리되면 지식 시스템이 실제 행동으로 연결되지 않는다.
- 모델 serving이나 agent workflow를 설계할 때 productionization과 adoption을 기술 배포 이후의 부가 작업이 아니라 core workflow로 포함해야 한다.

## 검증 필요 주장

- Palantir FDE의 공식 정의, Dev/FDE 구분 문구, Foundry 기능 설명은 Palantir 공식 자료로 재확인해야 한다.
- Airbus Skywise와 NYC Health + Hospitals 사례는 원문 글의 요약 주장으로만 다루고, 실제 성과와 계약 종료 맥락은 독립 자료로 대조해야 한다.
- "프로젝트의 70%가 problem framing에서 결정된다"는 표현은 정량 근거가 확인되지 않은 실무적 주장으로 둔다.
- 이 ingest에서는 첨부된 심층 리서치 파일은 별도로 캡처하지 않았다.
