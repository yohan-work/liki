# 하네스 엔지니어링

## 한 줄 정의

하네스 엔지니어링은 사람의 기억과 교육에만 의존하지 않고 규칙, 맥락, 절차, 도구, artifact, 검증을 작업 환경에 내장해 agent와 사용자가 일정 수준 이상의 결과를 반복해서 내도록 만드는 접근이다.

## 핵심 문제

- 조직 지식이 문서에 있어도 작업 중 발견하거나 적용하기 어렵다.
- 교육을 받아도 사용자별 경험과 의지에 따라 결과 편차가 크다.
- session이 끊기면 이전 결정, 요청, 결과물 위치를 잃기 쉽다.
- 숙련자의 암묵지가 개인에게 묶여 반복 작업과 onboarding 비용이 커진다.
- agent에게 목표만 주고 실행 규칙과 검증 기준을 주지 않으면 결과를 신뢰하기 어렵다.

## 작동 방식

- 항상 적용할 정책과 맥락은 repository instruction이나 steering 문서에 둔다.
- 특정 task의 반복 절차와 판단 기준은 [[Skills]]로 분리한다.
- directory, branch, README, 요청 번호, history log처럼 작업 상태가 남는 artifact 구조를 정의한다.
- script와 tool로 반복 action을 실행하고 lint, test, review로 결과를 검증한다.
- 사람이 승인해야 할 단계와 예외 처리 경로를 명시한다.
- 실제 실패와 rework를 반영해 instruction과 skill을 versioning한다.

[[Yanlog Kiro User Interview 003]]은 Kiro steering에 회사 정책과 FinOps 조사 구조를 넣고, 반복 action과 숙련자의 업무 방식을 skill로 만들어 onboarding과 작업 연속성을 높인 사례를 제시한다.

[[PyTorchKR AI ML Papers 2026 06 08 14]]가 소개한 `Self-Harness`는 하네스 자체를 개선 대상으로 다룬다. 에이전트가 과거 실행 추적에서 실패 패턴을 찾고 시스템 프롬프트와 운영 정책을 수정한다는 방향은 흥미롭지만, 개인 wiki나 coding workflow에 적용하려면 회귀 테스트, held-out task, 변경 diff, human approval을 함께 둬야 한다.

## 좋은 사용 사례

- coding agent가 repository 규칙을 따라 수정, 테스트, 기록까지 수행해야 하는 작업
- 조사 결과를 요청별 directory와 history로 보존하는 운영 분석
- 신규 구성원이 공통 workflow를 빠르게 시작해야 하는 onboarding
- 보안, 명명, 문서 형식처럼 반복적으로 누락되는 조직 표준
- 여러 session이나 agent가 같은 artifact를 이어서 다루는 장기 작업

## 실패 패턴

- 오래되거나 잘못된 규칙을 자동으로 모든 작업에 적용한다.
- instruction, skill, script에 같은 규칙을 복제해 precedence와 source of truth가 불명확하다.
- 형식 준수만 측정하고 실제 문제 해결 품질과 사용자 판단을 약화한다.
- 예외 처리와 human approval 없이 강제 규칙만 늘린다.
- artifact는 많이 만들지만 다음 session이 무엇을 읽고 이어갈지 정의하지 않는다.
- 숙련자의 결과만 복사하고 판단 조건과 실패 사례를 담지 않는다.

## 관련 개념과 차이

- [[Codex Harness]]는 Codex의 agent loop, thread, item, approval 같은 runtime 구조를 설명한다. 하네스 엔지니어링은 특정 제품 runtime보다 넓게 작업 환경과 조직 지식을 설계하는 실무 접근이다.
- [[Skills]]는 하네스를 구성하는 재사용 절차 단위다. 하네스는 skill 외에도 instruction, tool, artifact, log, validation, permission을 포함한다.
- [[Agentic Workflow]]는 agent가 목표를 수행하는 흐름이다. 하네스 엔지니어링은 그 흐름의 품질 하한선과 재현성을 높이는 환경 설계에 초점을 둔다.

## 예시

이 wiki repository의 `AGENTS.md`, `.codex/skills/wiki/SKILL.md`, templates, scripts, raw snapshot, source summary, index, log, lint는 ingest 결과를 반복 가능하고 검토 가능하게 만드는 하네스 구성요소다.

## 내 관점 / 임시 결론

좋은 하네스는 사용자의 사고를 대신하는 규칙 모음이 아니라, 반복 실수를 줄이고 판단이 필요한 지점을 더 선명하게 만드는 실행 환경이어야 한다. 품질은 문서 수가 아니라 적용 전후의 오류, rework, onboarding time, 검증 가능성으로 평가해야 한다.

## 열린 질문

- 어떤 규칙을 always-on instruction으로 두고 어떤 규칙을 task skill로 분리할 것인가?
- 하네스가 사용자 역량을 높이는지 단순히 도구 의존성을 만드는지 어떻게 구분할 것인가?
- 규칙 drift와 tool surface 차이를 어떤 자동 점검으로 발견할 수 있는가?
- 조직 공통 하네스와 개인별 customization 사이의 경계를 어떻게 정할 것인가?
- 하네스가 스스로 수정될 때 어떤 변경은 자동 적용하고 어떤 변경은 사람 승인 또는 decision record로 남겨야 하는가?
