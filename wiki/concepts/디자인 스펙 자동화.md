# 디자인 스펙 자동화

## 한 줄 정의

디자인 스펙 자동화는 design system component의 anatomy, API, property, token, structure, accessibility spec을 agent와 structured design data로 생성하고 유지하는 workflow다.

## 핵심 문제

Design system이 커질수록 component spec은 단순 문서가 아니라 engineering implementation의 기준점이 된다. [[Uber Automate Design Specs Article]]은 spec이 늦거나 부정확하면 engineer가 정의 대신 추정으로 구현하고, 그 결과 inconsistency와 tech debt가 생긴다고 설명한다.

## 자동화 대상

- Anatomy: component element를 번호 marker와 attribute table로 설명한다.
- API: configurable property, value, default, configuration example을 기록한다.
- Properties: variant axis와 boolean toggle preview를 보여준다.
- Color annotation: element와 state를 design token에 매핑한다.
- Structure: size, density, height, padding, spacing을 기록한다.
- Screen reader: VoiceOver, TalkBack, ARIA 같은 platform별 accessibility property를 정리한다.

## 작동 방식

1. Designer가 Figma component link와 agent가 design만으로 알 수 없는 context를 제공한다.
2. Agent skill이 section별 instruction, validation rule, schema, reference documentation을 불러온다.
3. [[Figma Console MCP]] 같은 bridge가 Figma component tree, variable, style, design token, parent-child relationship을 추출한다.
4. Agent가 token mapping, accessibility semantics, spec structure 같은 해석이 필요한 부분을 판단한다.
5. Programmatic rendering layer가 template을 가져와 table, text field, marker, cloned section을 Figma에 직접 채운다.
6. 생성된 spec은 design system의 기존 Figma file 안에서 검토된다.

Figma가 없거나 작은 PoC에서는 [[DESIGN.md]]를 먼저 두는 경량 경로도 가능하다. 이 경우 agent가 읽을 visual theme, semantic token, component state, responsive behavior, accessibility requirement와 금지 패턴을 Markdown contract로 version control한다. 다만 DESIGN.md는 관찰 기반 스타일 설명만으로 완성되는 source of truth가 아니며 실제 code·screenshot·interaction test와 함께 유지해야 한다.

## 좋은 사용 사례

- 여러 platform implementation stack이 같은 design system component를 공유하는 조직.
- Component, variant, token, accessibility spec 수가 많아 manual documentation이 drift되는 경우.
- Spec template과 validation rule이 이미 존재하고, agent가 채울 source of truth가 Figma에 구조화되어 있는 경우.
- Design token과 accessibility semantics를 engineer/designer가 함께 검토해야 하는 경우.

## 실패 패턴

- Screenshot만 보고 component spec을 만들게 한다.
- Figma layer 구조, token naming, variant modeling이 엉켜 있는데 agent에게 자동화를 맡긴다.
- Accessibility spec을 LLM 판단만으로 확정하고 platform reference나 expert review를 생략한다.
- Generated spec이 Figma file 안에 남더라도 changelog, validation, reviewer boundary가 없다.
- Local MCP라고 해서 model provider로 어떤 data가 전송되는지 확인하지 않는다.
- 외부 브랜드의 DESIGN.md를 그대로 복사해 trademark, proprietary font, image asset과 고유 visual identity 경계를 무시한다.
- DESIGN.md가 존재한다는 이유만으로 agent가 자동으로 읽고 구현한다고 가정한다.

## 내 관점 / 임시 결론

이 개념의 핵심은 "문서를 자동으로 써준다"가 아니라 "design source of truth를 읽고, 정해진 template과 validation rule 안에서, 원래 작업 공간에 검토 가능한 spec artifact를 만든다"는 점이다. 개인 PoC로 가져오려면 Figma MCP부터 붙이기보다, 작은 component 1개에 대해 token mapping과 structure spec만 생성하고 사람이 검증하는 흐름이 현실적이다.

## 열린 질문

- LLM Wiki에도 source summary section별 skill을 두면 유지보수 품질이 올라갈까?
- Figma component layer naming과 token 구조가 나쁘면 agent spec 생성 품질은 어디서 먼저 깨지는가?
- Design spec 자동화 결과를 CI나 accessibility test와 연결할 수 있는가?
- Figma, code token, DESIGN.md 중 source of truth와 generated projection을 어떻게 선언할 것인가?
