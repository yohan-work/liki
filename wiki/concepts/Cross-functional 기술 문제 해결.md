# Cross-functional 기술 문제 해결

## 한 줄 정의

Cross-functional 기술 문제 해결은 여러 팀과 기술 영역의 경계에 남은 모호한 문제를 재정의하고, 결정권·책임·의존성·완료 기준이 보이는 실행 구조로 바꾸는 작업이다.

## 핵심 문제

조직이 커지면 제품, 인프라, 데이터, 보안, 운영처럼 여러 영역에 걸친 문제가 늘어난다. 각 팀이 자기 역할을 수행해도 전체 문제를 끝까지 책임지는 owner가 없거나, 결정 구조와 우선순위가 충돌해 상태 공유만 반복되는 회색지대가 생긴다.

이때 회의 횟수, 리스크 표, 일정 보고를 늘리는 것은 가시성을 높일 수 있지만 문제 자체를 해결 가능한 단위로 바꾸지는 못한다.

## 진단 질문

- 지금 보이는 현상 아래의 반복되는 병목은 무엇인가?
- 팀 단위 문제인가, 여러 팀의 책임 경계에서 생긴 문제인가?
- 어떤 결정이 비어 있고 누가 그 결정을 내려야 하는가?
- 실질 owner가 누구이며 책임을 맡을 권한과 context가 있는가?
- 어떤 의존성과 우선순위 충돌이 실행을 막고 있는가?
- 완료를 판단할 결과와 증거는 무엇인가?
- 임시 해결 후 ownership과 운영 책임을 어디로 이전할 것인가?

## 실행 구조

1. 현상과 원인을 분리하고 문제를 다시 정의한다.
2. 이해관계자, 시스템, 의존성, 제약을 한 화면에 드러낸다.
3. 결정할 항목과 선택지를 명시한다.
4. 결정권자, 실행 owner, 협력자, 검토자를 구분한다.
5. 먼저 제거할 blocker와 action 순서를 정한다.
6. 완료 기준과 검증 evidence를 합의한다.
7. 실행 후 반복 병목이 사라졌는지 확인하고 영구 ownership을 정한다.

[[Toss Tech Cross Functional TPM Tip Article]]은 이 과정을 토스 TPM의 핵심 역할로 설명하며, 문제 재정의, 구조화, 전략적 판단, 실행 전환, 권한 없는 영향력, 사람과 구조를 함께 보는 능력을 강조한다.

## 필요한 역량

- 문제 재정의: 일정 지연이나 소통 부족 같은 증상 아래의 구조적 원인을 찾는다.
- 구조화: 복잡한 맥락을 결정, 책임, 의존성, 순서로 나눈다.
- 중요도 판단: 일시적 조정과 반복되는 구조적 문제를 구분한다.
- 실행 전환: 논의를 실제 결정과 action plan으로 닫는다.
- 맥락 번역: 서로 다른 직군과 팀의 언어, 제약, 성공 기준을 연결한다.
- 권한 없는 영향력: 직함보다 신뢰, 판단, 실적으로 참여와 결정을 이끌어낸다.
- 조직 설계 관점: 프로세스뿐 아니라 역할과 팀 구조의 변경 필요성도 본다.

## 실패 패턴

- 회의와 보고의 양을 문제 해결 성과로 착각한다.
- 모든 이해관계자의 완전한 합의를 기다리며 결정 책임을 흐린다.
- 임시 coordinator가 영구 owner처럼 모든 실행 부담을 떠안는다.
- 문제를 크게 정의한 뒤 작은 결정과 action으로 분해하지 못한다.
- 공식 권한 없이 개입하면서 각 팀의 기존 책임과 전문성을 침범한다.
- 역할 철학을 먼저 선언하고 작동 사례와 신뢰를 만들지 않는다.
- 단기 blocker만 제거하고 반복 원인이 되는 역할·운영 구조를 그대로 둔다.

## 자율성이 낮은 조직에서의 시작

- 작지만 반복되는 병목 하나를 해결해 신뢰를 만든다.
- 기존 회의에서 상태 공유와 별도로 결정할 항목을 명시한다.
- 작은 이니셔티브 안에서 owner, 결정권자, 완료 기준을 합의한다.
- 역할 설명보다 공백을 메우고 실행을 끝낸 사례를 기록한다.
- 이상적인 조직 구조를 한 번에 요구하지 않고 검증된 범위를 넓힌다.

## 관련 개념과 차이

- [[Orchestration]]: tool과 agent workflow를 조율하는 개념이다. Cross-functional 문제 해결은 사람과 조직의 결정·책임 구조까지 포함하는 더 넓은 실행 문제다.
- [[Forward Deployed Engineer]]: 현장 문제를 기술 구현과 adoption으로 연결하는 역할 패턴이다. Cross-functional 문제 해결은 구현 주체와 무관하게 팀 경계의 owner·decision 공백을 다룬다.
- [[Human-in-the-loop]]: 중요한 결정과 승인 지점을 사람에게 남긴다. Cross-functional 문제 해결은 그 사람이 누구이며 어떤 근거로 결정할지를 선명하게 만든다.
- [[Personal Operating System]]: 개인 규모에서도 여러 프로젝트, source, agent, decision 사이의 공백을 진단하는 데 같은 질문을 적용할 수 있다.

## 개인 운영체계 적용

개인 workflow에는 여러 팀이 없더라도 역할 간 경계가 존재한다. 자료 수집자, 분석자, 결정자, 실행자, 검토자 역할이 분리되지 않으면 ingest는 늘지만 실행되지 않거나, 아이디어는 쌓이지만 owner와 완료 기준이 없는 상태가 된다.

정체된 작업에는 새 목록을 추가하기 전에 문제, 결정, owner, dependency, next action, done evidence 여섯 항목을 먼저 기록하는 편이 낫다.

## 내 관점 / 임시 결론

Cross-functional 문제 해결의 핵심 산출물은 더 정교한 상태 보고가 아니라 누가 무엇을 왜 결정하고 어떤 evidence로 완료를 판단하는지 보이는 실행 구조다. 조율은 필요하지만 문제 재정의와 책임 구조를 만들지 못하면 관리 활동으로 끝난다.

## 열린 질문

- 임시로 공백을 메우는 역할과 기존 조직 owner의 책임을 어떻게 구분할 것인가?
- 여러 팀의 목표가 실제로 충돌할 때 우선순위 결정권자는 누가 되어야 하는가?
- 개인 운영체계의 정체 작업에 이 진단 질문을 적용하면 실행률이 높아지는가?
- 문제 구조화 문서가 또 하나의 보고서가 되지 않게 하는 최소 형식은 무엇인가?
