# 토스증권 API 투자 운영 시스템

## 현재 판단

- 상태: candidate
- 판단 이유: [[토스증권 Open API]]가 시장 데이터, 계좌, 보유 자산, 주문 가능 정보, 주문 실행을 REST API로 제공하므로 개인 투자 workflow를 외부 소프트웨어로 끌어낼 수 있다. 다만 실제 주문 자동화는 리스크가 크므로 첫 검증은 read-only 투자 운영 리포트와 주문 전 체크리스트에 제한한다.
- 다시 볼 조건: 토스증권 Open API 사용 신청 조건, 약관, 자동주문 허용 범위가 확인되고, read-only 리포트가 1주 이상 실제 의사결정 시간을 줄이면 MVP로 승격한다.

## 원형 메모

토스증권이 API를 제공한다는 것은 증권 앱 안에 갇혀 있던 시장 데이터, 계좌 상태, 보유 종목, 주문 가능 정보, 주문 기능을 개인 소프트웨어와 AI workflow에 연결할 수 있다는 뜻이다. 단순 자동매매 봇보다, 매일 반복하는 확인, 판단, 기록, 실수 방지를 자동화하는 개인 투자 운영 시스템이 더 적합하다.

## 한 줄 요약

토스증권 Open API를 이용해 관심 종목, 보유 자산, 환율, 장 운영 시간, 매수/매도 가능 정보를 자동으로 모으고, 투자 판단 리포트와 주문 전 체크리스트를 제공하는 개인 투자 운영 시스템을 만든다.

## 문제

개인 투자자는 매일 여러 정보를 반복해서 확인한다. 보유 종목 손익, 관심 종목 가격, 환율, 장 운영 시간, 현금 여력, 매도 가능 수량, 수수료, 주문 상태를 각각 앱 화면에서 확인해야 한다. 이 과정은 시간이 들고, 장이 열려 있을 때는 감정적 판단이나 착오 주문으로 이어지기 쉽다.

기존 증권 앱은 거래 실행에는 강하지만, 개인의 장기적 운영 루틴에는 약하다. 내가 왜 관심 종목을 보고 있는지, 어떤 조건이 충족되었는지, 어제와 오늘 포트폴리오 상태가 어떻게 달라졌는지, 주문 전에 무엇을 확인해야 하는지를 내 기준으로 구조화하기 어렵다.

## 기대 효과

- 토스증권 계좌를 단순 매매 앱이 아니라 개인 투자 backend로 사용할 수 있다.
- 매일 반복하는 확인 작업을 자동 리포트로 줄인다.
- 관심 종목과 보유 종목을 같은 화면에서 비교한다.
- 환율과 장 운영 시간을 투자 판단 context로 자동 반영한다.
- 실제 주문 전 실수 가능성을 줄이는 guardrail을 만든다.
- 레버리지 ETF처럼 일반 주식과 다른 위험 구조를 가진 상품을 별도 경고로 표시한다.
- LLM은 매매 결정을 대신하지 않고, 데이터 요약과 질문 답변, 체크리스트 설명에만 사용한다.

## 내가 끌린 이유

이 아이디어는 [[Personal Operating System]]과 잘 맞는다. 투자 판단은 뉴스, 가격, 계좌 상태, 개인 규칙, 실행 기록이 모두 섞이는 영역이다. API가 제공되면 이 데이터를 사람이 매번 앱에서 확인하지 않고, 개인 운영 루틴 안으로 가져올 수 있다.

또한 [[Agentic Workflow]]의 좋은 실험 사례다. agent에게 바로 주문 권한을 주는 것이 아니라, read-only data retrieval, rule-based calculation, human confirmation, execution boundary를 분리하는 설계가 필요하기 때문이다.

## 방향성 적합성

- [[LLM Wiki]]: 토스증권 API 문서, 투자 아이디어, 실행 로그를 장기 지식으로 축적할 수 있다.
- [[Personal Operating System]]: 매일 반복되는 투자 확인 workflow를 자동화 후보로 다룰 수 있다.
- [[Quant Trading]]: 시장 데이터, 계좌 상태, 주문 실행 layer를 분리해 작은 PoC로 검증할 수 있다.
- [[Human-in-the-loop]]: 실제 주문은 사람 승인 뒤에 두고, 시스템은 근거와 위험 신호를 제시한다.

## 대상 사용자

- 1차 사용자: 토스증권을 쓰고, 국내/미국 주식을 함께 보며, 매일 계좌와 관심 종목을 확인하는 나 자신.
- 2차 사용자 후보: 자동매매보다는 투자 루틴, 리밸런싱, 기록, 실수 방지에 관심 있는 개인 투자자.
- 제외할 사용자: 초단타, 고빈도 매매, 완전 자동매매를 원하는 사용자, 투자 자문 서비스를 기대하는 사용자.

## 사용 맥락

- 장 시작 전: 오늘 한국/미국 장 운영 여부, 보유 종목 상태, 환율, 관심 종목 조건 충족 여부를 확인한다.
- 장중: watchlist 조건 알림, 급등/급락, 매수 유의사항, 매수 가능 금액을 확인한다.
- 주문 전: 주문 가능 금액/수량, 수수료, 장 운영 시간, 목표 비중, 최대 주문 한도, 중복 주문 가능성을 확인한다.
- 장 종료 후: 하루 손익, 조건 충족 종목, 놓친 주문 후보, 실제 판단 기록을 정리한다.
- 주간 리뷰: 포트폴리오 비중, 반복된 실수, 전략 후보의 성과를 검토한다.

## 해결 가설

토스증권 Open API의 read-only 데이터만으로도 개인 투자자의 반복 확인 시간을 줄이고, 투자 판단의 품질을 높일 수 있다. 실제 주문 자동화는 나중 단계이며, 첫 가치는 자동 실행이 아니라 운영 가시성, 조건 감지, 기록, 실수 방지에서 나온다.

초기 제품은 다음 5개 모듈로 나눈다.

- 시장 데이터 수집기: `stocks`, `prices`, `orderbook`, `trades`, `price-limits`, `candles`를 사용해 관심 종목 상태를 수집한다.
- 시장 context 수집기: `exchange-rate`, `market-calendar/KR`, `market-calendar/US`로 환율과 장 운영 시간을 확인한다.
- 계좌 상태 수집기: `accounts`, `holdings`, `buying-power`, `sellable-quantity`, `commissions`로 보유 자산과 거래 가능 정보를 확인한다.
- 판단 보조 엔진: watchlist 조건, 목표 비중, 리밸런싱 필요 여부, 주문 전 체크리스트를 계산한다.
- 위험 패널: 레버리지 ETF, 변동성 완화장치, 매수 유의사항, 급격한 drawdown 같은 상품별 위험 신호를 분리해 보여준다.
- 리포트/저널: 매일 요약, 주문 후보, 실제 판단, 결과 회고를 Markdown 또는 dashboard로 저장한다.

## 만들 수 있는 구체적 아이디어

### 개인 투자 운영 리포트

매일 아침과 저녁에 자동으로 계좌와 관심 종목을 요약한다.

- 보유 종목 평가손익
- 현금과 매수 가능 금액
- 환율 변화와 원화 평가 영향
- 오늘 한국/미국 장 운영 여부
- 관심 종목 가격 변화
- 전일 대비 크게 움직인 종목
- 확인해야 할 매수 유의사항
- 오늘 주문 후보와 보류 이유

### Watchlist Intelligence

관심 종목을 단순 가격 목록이 아니라 조건 기반 상태판으로 만든다.

- 목표 가격 도달
- 최근 평균 대비 급등/급락
- 거래량 급증
- 상/하한가 근접
- 매수 유의사항 발생
- 미국장 프리/애프터 시간대 움직임
- 환율 기준 매수 부담 변화
- 레버리지 ETF의 경우 [[레버리지 ETF 변동성 끌림]] 경고와 최근 변동성 구간 표시

### 포트폴리오 리밸런싱 보조 도구

목표 비중과 현재 비중을 비교해 매수/매도 후보를 계산한다.

- 현재 보유 비중
- 목표 비중 대비 차이
- 추가 매수 또는 일부 매도 필요 금액
- 매수 가능 금액 기준 실행 가능성
- 세부 주문은 생성하지 않고 주문 초안만 제안

### AI 투자 비서

LLM을 붙이되 주문 권한은 주지 않는다. LLM은 데이터를 설명하고 질문에 답한다.

- "오늘 내 계좌에서 가장 크게 움직인 종목은?"
- "미국 주식 비중이 얼마나 돼?"
- "환율 때문에 원화 평가액이 얼마나 변했을 가능성이 있어?"
- "이번 주 watchlist 중 조건을 충족한 종목은?"
- "내 현금 기준으로 가능한 주문 후보만 계산해줘."

### 주문 전 체크리스트 자동화

실제 주문을 넣기 전에 실수 가능성을 점검한다.

- 장 운영 시간 확인
- 매수 가능 금액 확인
- 판매 가능 수량 확인
- 수수료 포함 예상 비용 확인
- 주문 금액이 개인 한도를 넘는지 확인
- 시장가 주문 시 변동성이 큰 상태인지 확인
- 레버리지 ETF/ETN이면 일일 레버리지 상품 특성과 volatility decay를 확인했는지 표시
- 1억원 이상 주문 확인 플래그 필요 여부 확인
- `clientOrderId` 중복 가능성 확인

### Paper Trading / Dry-run 엔진

실제 주문 API를 쓰기 전에 가상 주문장을 만든다.

- 실제 시세 기반 가상 매수/매도
- 전략 후보별 성과 추적
- 실제 계좌와 가상 계좌 비교
- 주문 가능 금액과 수량을 반영한 현실적 simulation
- 일정 기간 검증 후 실제 주문 후보로 승격

### 시계열 예측 연구 패널

[[TimesFM]] 같은 모델은 주문 자동화가 아니라 read-only research panel에서 먼저 검증한다.

- 가격보다 수익률, 변동성, 거래량, 거시경제 지표 중 target을 먼저 명시한다.
- seasonal naive와 단순 statistical model을 baseline으로 둔다.
- walk-forward split으로 leakage를 막고 regime별 error를 분리한다.
- point forecast뿐 아니라 quantile coverage와 calibration을 기록한다.
- transaction cost와 turnover 반영 전에는 예측 성능을 투자 성과로 해석하지 않는다.
- 모델 출력은 주문 후보가 아니라 검토할 research signal로만 표시한다.

### 투자 저널

매수/매도 판단과 결과를 장기 기록으로 남긴다.

- 주문 전 의도 기록
- 주문 후보의 조건 충족 이유
- 실제 주문 여부와 보류 이유
- 며칠 뒤 결과 자동 추적
- 반복되는 손실 패턴 분석
- 종목별 의사결정 히스토리 생성

## 기존 대안

- 토스증권 앱: 실제 매매와 계좌 확인에는 적합하지만, 개인별 조건, 자동 리포트, 장기 회고, AI 질의응답 workflow는 제한적이다.
- 스프레드시트: 자유도는 높지만 API 연동, 자동 갱신, 장 운영 시간, 주문 가능 정보와 연결하기 어렵다.
- 일반 포트폴리오 앱: 여러 증권사 통합에는 강할 수 있지만, 토스증권 주문 가능 정보와 주문 전 체크리스트까지 깊게 연결되지는 않는다.
- 자동매매 봇: 실행력은 있지만 실수 방지, 설명 가능성, human approval boundary가 약하면 개인 workflow에는 위험하다.

## 기존 대안의 부족한 점

기존 도구들은 대체로 "보기"와 "거래"에 집중한다. 하지만 개인에게 필요한 것은 매일 반복되는 확인을 줄이고, 판단 기준을 일관되게 적용하며, 나중에 왜 그런 결정을 했는지 되돌아볼 수 있게 만드는 운영 체계다.

## 더 일반화한 문제

증권 API를 통해 해결하려는 진짜 문제는 자동매매가 아니라, 개인의 반복적 금융 의사결정을 데이터 기반 운영 루틴으로 바꾸는 것이다. 이 문제는 투자뿐 아니라 예산, 저축, 세금, 보험, 구독 관리 같은 개인 재무 workflow에도 확장될 수 있다.

## 더 작게 줄인 형태

첫 버전은 주문 기능을 제외한 read-only 리포트다.

- 관심 종목 10개
- 보유 종목 전체
- 환율
- 한국/미국 장 운영 시간
- 매수 가능 금액
- 매일 1회 Markdown 리포트 생성
- LLM 요약은 선택 사항
- 실제 주문, 정정, 취소는 제외

## 빠른 검증 방법

1주 검증은 "토스증권 read-only 투자 운영 리포트"로 제한한다.

1. 토스증권 API 인증과 read-only endpoint 호출을 확인한다.
2. 관심 종목 10개를 정한다.
3. 보유 종목과 관심 종목을 매일 1회 수집한다.
4. 환율과 장 운영 시간을 함께 저장한다.
5. Markdown 리포트를 자동 생성한다.
6. 7일 동안 실제로 리포트를 읽고, 앱을 직접 확인하는 시간이 줄었는지 기록한다.

## 검증 질문

- 매일 증권 앱을 여는 횟수가 줄어드는가?
- 리포트를 보면 5분 안에 계좌 상태와 관심 종목 상태를 파악할 수 있는가?
- 자동 리포트가 실제 투자 판단 또는 보류 판단에 도움을 주는가?
- 환율과 장 운영 시간이 함께 보일 때 미국주식 판단이 더 쉬워지는가?
- watchlist 조건 알림이 잡음보다 유용한 신호를 더 많이 주는가?
- 주문 전 체크리스트가 실제 실수 가능성을 줄일 수 있는가?
- LLM 요약이 숫자/근거를 왜곡하지 않고 설명만 잘 보조하는가?
- 레버리지 ETF 보유 또는 watchlist 종목에 대해 volatility decay 위험을 이해하기 쉽게 표시하는가?

## 예상 난이도

- 기술 난이도: 중간. OpenAPI JSON이 있어 client 생성과 schema 확인은 수월하지만, 인증, 계좌 헤더, rate limit, error handling, secret 관리가 필요하다.
- 제품 난이도: 중간. 어떤 정보를 매일 볼 가치가 있는지 좁히는 것이 핵심이다.
- 운영 난이도: 중간. market data polling, rate limit, 장 운영 시간, 환율 기준 시각을 관리해야 한다.
- 리스크 난이도: 높음. 주문 API까지 확장하면 실제 자산 손실 가능성이 있으므로 human confirmation과 hard limit이 필수다.

## MVP 승격 판단

- 승격 후보: 토스증권 read-only 투자 운영 리포트 7일 PoC
- 필요한 선행 작업: API 사용 신청 조건 확인, credential 발급, 약관/자동주문 허용 범위 확인, secret 저장 방식 결정, 관심 종목 10개 선정.
- 성공 기준 초안: 7일 중 5일 이상 리포트를 읽고, 하루 5분 안에 계좌/관심 종목 상태를 파악하며, 증권 앱 직접 확인 시간이 줄어든다.
- 제외할 범위: 실제 주문 생성, 주문 정정, 주문 취소, 완전 자동매매, 투자 자문, 종목 추천, 뉴스 크롤링, 다중 증권사 연동.

## 승격 조건

- read-only endpoint 호출이 안정적으로 동작한다.
- 계좌와 관심 종목 데이터가 하나의 리포트에서 의미 있게 결합된다.
- 리포트가 실제 확인 시간을 줄인다.
- 주문 없이도 충분한 사용 가치가 확인된다.
- 주문 전 체크리스트가 별도 모듈로 분리될 만큼 반복적인 필요가 확인된다.

## 보류 조건

- 토스증권 API 사용 신청 또는 약관상 개인 PoC 운영이 어렵다.
- API 호출 안정성이나 rate limit 때문에 매일 리포트 품질이 낮다.
- 리포트를 봐도 기존 앱 확인 습관을 대체하지 못한다.
- 투자 판단을 자동화하려는 유혹이 커져 risk boundary가 흐려진다.
- 계좌/투자 정보의 민감도 때문에 로컬 저장과 secret 관리 부담이 과도하다.

## 기각 조건

- API 사용 조건이 개인 자동화와 맞지 않는다.
- read-only 리포트가 1주 검증에서 반복 사용 가치를 만들지 못한다.
- 주문 자동화 없이는 가치가 없다는 결론이 나오는데, 주문 자동화 리스크를 감당할 이유가 없다.
- 금융 데이터 저장과 보안 운영 부담이 개인용 도구 가치보다 크다.

## 주문 자동화 경계

실제 주문 API는 마지막 단계로 둔다. 자동 주문을 검토하더라도 다음 조건이 없으면 진행하지 않는다.

- dry-run mode가 기본값이다.
- 1회 주문 최대 금액과 일일 총 주문 한도가 코드에 hard-coded 또는 config로 고정된다.
- 주문 생성 전 `buying-power`, `sellable-quantity`, `commissions`, market calendar를 확인한다.
- 모든 주문은 사람이 최종 승인한다.
- `clientOrderId`를 필수로 사용한다.
- 주문 후 `getOrder`로 상태를 확인한다.
- 실패, 중복, 부분체결, 취소 실패를 별도 상태로 기록한다.
